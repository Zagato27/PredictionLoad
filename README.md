## Сервис прогнозирования нагрузки и задержек (Flask + Jinja2 + Plotly + Pydantic)

Полноценный веб‑сервис, который принимает результаты ступенчатого нагрузочного теста (RPS, avg_ms, max_ms, утилизации) и строит человекочитаемые прогнозы по задержкам (средним и максимальным) и утилизации ресурсов, а также рекомендации по масштабированию.  
Интерактивные графики отрисовываются в UI с помощью Plotly.js; серверные шаблоны на Jinja2; валидация входа — Pydantic; математика и подгонка моделей — numpy/pandas/scipy.

### Ключевые возможности
- Модели: M/M/1, M/M/c (Erlang C), Kingman (G/G/1), USL (Universal Scalability Law).
- Оценка сервисного времени S и устройств D_i (CPU/IO/NET) по линейному участку (утилизация vs RPS).
- Прогноз задержек при целевом `target_rps`, проверка SLO по максимальной задержке.
- Рекомендации по масштабированию: инстансы по CPU/IO/NET и подбор `c` для M/M/c.
- Интерактивные графики: latency vs RPS (+линия SLO, вертикаль «Насыщение (USL)»), utilization vs RPS, instances vs RPS.
- Экспорт PDF отчёта (опционально через WeasyPrint).
- REST API (`POST /api/forecast`) и auto OpenAPI (`GET /openapi.json`).

1) Требования: Python 3.10+ (рекомендуется 3.11), macOS/Linux/WSL.  
2) Установка:
```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -U pip
pip install -r requirements.txt
```
3) Запуск в dev‑режиме:
```bash
export DEBUG=1
python app.py
# или через gunicorn (как в проде):
# gunicorn -w 4 -b 0.0.0.0:8000 app:app
```
4) Откройте браузер: `http://127.0.0.1:8000`

5) На главной странице вставьте JSON (пример: `data/sample.json`), задайте `target_rps` и нажмите «Рассчитать прогноз».

Примечание (PDF на macOS): для WeasyPrint могут понадобиться системные библиотеки (Pango, Cairo). На macOS можно установить через `brew install pango cairo`. Если не требуется PDF, WeasyPrint можно пропустить.

### Описание графиков и таблиц в отчёте

1) Таблица «Модели и прогнозы (мс)»  
- «Средняя задержка (мс)»: прогноз среднего времени отклика при целевом RPS согласно каждой модели.  
- «Максимальная (мс)»: пиковая задержка, оценивается как средняя × коэффициент (отношение `max_ms/avg_ms` базовой ступени).  
- «Рекомендация»: краткий совет по применимости модели (например, риск невыполнения SLO, необходимость увеличить `c` в M/M/c).

2) Таблица «Параметры и оценки»  
- `S (мс)`: базовое сервисное время. Оценивается через максимальный из наклонов утилизаций по линейному участку: `S ≈ max(D_i)`.  
- `α (USL)`, `β (USL)`: параметры универсального закона масштабируемости. `α` отражает накладные расходы (синхронизация и т.п.), `β` — эффект координации/ретраев.  
- `Ca`, `Cs`: коэффициенты вариации для входного потока и сервиса; используются в формуле Кингмана для оценки очереди.

3) График «Задержка vs RPS»  
- Точки Observed — фактические средние и максимальные задержки на ступенях теста.  
- Линии M/M/1, USL, G/G/1 — прогнозы средних задержек по моделям на непрерывной шкале RPS.  
- Вертикальная пунктирная линия — целевой RPS.  
- Вертикальная красная пунктирная — «Насыщение (USL)» (где ожидается пик масштабируемости по USL).  
- Горизонтальная линия SLO — порог по максимальной задержке (если задан в `target.slo_ms_max_optional`). Наводите курсор для подсказок (hover).

4) График «Утилизация vs RPS»  
- Линии CPU/IO/NET — прогноз утилизаций ресурсов по формулам `U_i = X · D_i`.  
- Анализируйте, где утилизации пересекают пороги `u_max_*` из раздела `capacity` — это зоны, где потребуется масштабирование.

5) График «Утилизация RAM vs RPS»  
- Линия RAM — прогноз утилизации памяти `U_ram = X · D_ram`, где `D_ram` оценивается регрессией по ступеням.  
- Горизонтальная линия `u_max_ram` — допустимый порог. Точки выше порога подсвечены.  
- Используйте график для оценки момента, когда потребуется масштабирование из‑за памяти.

6) График «Инстансы vs RPS»  
- Ступенчатые линии показывают необходимое количество инстансов по каждой метрике (CPU/IO/NET), исходя из текущих порогов `u_max_*`.  
- Изломы кривых соответствуют событиям масштабирования (увеличения числа инстансов). Наведение курсора показывает численные значения.

### Формат входного JSON (кратко)
Kubernetes-поля (см. `data/sample.json`):
```json
{
  "steps": [
    {
      "step": "s1",
      "rps": 62,
      "avg_ms": 540,
      "max_ms": 586,
      "pods": 2,
      "cpu_usage_m": 520,
      "cpu_request_m_per_pod": 260,
      "cpu_limit_m_per_pod": 1200,
      "mem_workingset_mib": 2380,
      "mem_request_mib_per_pod": 1290,
      "mem_limit_mib_per_pod": 1400,
      "net_in_mbps": 12.4,
      "net_out_mbps": 8.7,
      "concurrency_optional": 33,
      "errors_pct": 0.0
    }
    // ... минимум 2 ступени
  ],
  "target": {
    "target_rps": 150,
    "slo_ms_max_optional": 200
  },
  "capacity": {
    "u_max_cpu": 0.7,
    "u_max_ram": 0.7,
    "net_capacity_in_mbps_optional": 200,
    "net_capacity_out_mbps_optional": 200,
    "mmc_c_optional": 4
  },
  "modeling": {
    "use_m_m_1": true,
    "use_m_m_c": true,
    "use_kingman": true,
    "use_usl": true
  }
}
```
Ограничения: `rps>0`, `avg_ms>0`, `max_ms>=avg_ms`, утилизации `0..1`. Поля с постфиксом `_optional` — опциональны.

### API
- `GET /` — форма загрузки JSON, поля `target_rps`, `slo_ms_max_optional`.

### Модели и постановка
- Закон Литтла: \(N = X \cdot R\), где \(R\) — средняя задержка (ms переведённые в секунды).
- M/M/1: \(R = \frac{S}{1 - X S}\), \(\rho = XS < 1\). Максимальная задержка аппроксимируется коэффициентом \(\frac{max\_ms}{avg\_ms}\) базовой ступени.
- M/M/c (Erlang C): \(R = \frac{C(c,a)}{c\mu - \lambda} + \frac{1}{\mu}\), \(a=\lambda/\mu\).
- Kingman (G/G/1): \(W_q \approx \frac{\rho}{1-\rho} \cdot \frac{C_a^2 + C_s^2}{2} \cdot S\), \(R \approx S + W_q\).
- USL: \(X(N) = \dfrac{X_1 N}{1 + \alpha(N-1) + \beta N(N-1)}\), подгонка по точкам \((N, X)\), где \(N = X \cdot avg\_s\).

Оценка параметров:
- Линейный участок берётся из ступеней с `errors_pct ≤ 5%` и умеренным ростом `avg_ms` (<20% между соседними).
- \(D_i\) оцениваются регрессией `util_i ~ rps` через начало координат; \(S \approx \max(D_i)\).
- \(C_a\) — коэффициент вариации RPS между ступенями; \(C_s\) — средняя относительная разница `(max_ms-avg_ms)/avg_ms` (обрезана до разумных границ).
- USL подгоняется через `scipy.optimize.curve_fit`, стартовые значения и ограничение параметров устойчивы к шуму.

### Тонкости и рекомендации
- Входной JSON ограничен 1 MB (см. `app.config["MAX_CONTENT_LENGTH"]`). Соблюдайте экономность.
- Старайтесь, чтобы ступени были как минимум в двух областях нагрузки (низкая и средняя/высокая) — это улучшает регрессию и USL.
- Если доля ошибок `errors_pct > 5%`, ступени исключаются из визуализации и линейной регрессии, но могут учитываться для общих диапазонов.
- WeasyPrint требует системных библиотек (Pango/Cairo); используйте контейнер из `Dockerfile`, чтобы избежать проблем окружения.

---

### Частые проблемы (FAQ)
- «max_ms < avg_ms» — проверьте корректность метрик. Максимум должен быть ≥ среднему.
- «Недостаточно steps для регрессии — минимум 2» — добавьте хотя бы две валидные ступени.
- «Внутренняя ошибка: ...» — включите `DEBUG=1`, посмотрите логи в консоли; проверьте корректность JSON и типы полей.

## Сервис прогноза нагрузки (Flask + Jinja2 + Plotly)

Веб‑сервис принимает результаты ступенчатого нагрузочного теста в формате JSON и строит человекочитаемые прогнозы по задержкам и утилизации ресурсов. UI рендерится серверными шаблонами Jinja2, графики — через Plotly.js, стили — Bootstrap 5. Валидация входа — Pydantic. Для подгонки моделей используются numpy/scipy/pandas.

### Быстрый старт (локально)
1. Установите зависимости:
   ```bash
   python3 -m venv .venv && source .venv/bin/activate
   pip install -r requirements.txt
   ```
2. Запуск в dev-режиме:

### REST API
- POST `/api/forecast` — вход `application/json` (схема см. `/openapi.json`).
- GET `/healthz` — health-check.
- GET `/openapi.json` — спецификация API (OpenAPI 3.0).

### Основные возможности
- Поддержка моделей: M/M/1, M/M/c (Erlang C), Kingman (G/G/1), USL (Universal Scalability Law).
- Оценка `S` и `D_i`: регрессия утилизаций от RPS на линейном участке.
- Прогнозы по целевому `target_rps`, проверка SLO (макс. задержка).
- Интерактивные графики: задержка vs RPS, утилизация, число инстансов.
- Экспорт PDF (опционально, через weasyprint).
